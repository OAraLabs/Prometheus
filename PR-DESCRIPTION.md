## Bugs found & fixed on the way (each required for acceptance item 1 to be true)

- **Slack `start()` hung the daemon forever — even with VALID tokens.** `gateway/slack.py` awaited slack-bolt's `AsyncSocketModeHandler.start_async()`, which is documented as "starts infinite sleep to prevent the termination of this process": nothing after the Slack block (Discord, heartbeat, web server) ever ran when Slack was enabled. Never caught live — only ever exercised over doubles. Now: `auth.test` validates the bot token and `apps.connections.open` validates the app token — both raise loudly on bad auth (→ the daemon's try/except logs and continues) — then the connect goes to a background task (the same non-blocking start the Discord adapter uses; the Discord comment even *assumed* Slack already did this). Regression-pinned by `TestSlackStartNonBlocking` (start returns within a timeout, connects in background, bad bot/app token each raise).
- **Telegram startup failure killed the daemon.** The Telegram construction block was the only unguarded one — a bad token raised out of `telegram.start()` and took the whole daemon down. Now try/except'd exactly like the Slack/Discord blocks (log + continue; approval-queue wiring skipped when the adapter failed).
- **Token leaks in the daemon log.** (a) python-telegram-bot's `InvalidToken` message embeds the token value — the gateway-startup error logs now redact the token substring in all three blocks. (b) httpx logs every request URL at INFO, and the Telegram Bot API puts the token in the URL path (`…/bot<token>/getMe`) — so any daemon with Telegram enabled was logging its real token on every Telegram API call. The daemon now sets the `httpx` logger to WARNING (the doctor CLI already did this for its own probes).
