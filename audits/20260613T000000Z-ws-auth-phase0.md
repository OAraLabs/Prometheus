# Phase A — WS Auth: Phase 0 Survey + Mechanism Decision

**Branch:** `feat/ws-auth` (off `main @ 3f6c29b`, clean, 0/0 vs origin)
**Date:** 2026-06-13 · **Scope:** authenticate the WebSocket bridge on :8010 (currently fully open).

## Where the WS server is created

`src/prometheus/web/ws_server.py` — `WebSocketBridge`. Started via
`websockets.serve(self._handler, host, port)` at [ws_server.py:53](src/prometheus/web/ws_server.py#L53),
default `port=8010`. Constructed in `launch_web` at
[launcher.py:80](src/prometheus/web/launcher.py#L80); both REST (:8005) and WS (:8010) launched
together by the daemon ([daemon.py:1211](src/prometheus/daemon.py#L1211)). Runtime: system
`python3`, `websockets==15.0.1`.

**Today there is NO auth on :8010.** `_handler` adds the socket to `_clients`, immediately sends a
`connected` welcome frame, and forwards every SignalBus event to every client. Anyone who can
reach the port receives everything.

## Every event type streamed (the exposure)

From `_handler` / `_run_agent` / `_on_signal`: `connected`, `subscribed`, `chat_message`,
`chat_delta`, `tool_call_start`, `tool_call_end`, `chat_done`, `agent_state`, `error`,
`sentinel_signal`, `dream_start`, `dream_phase`, `dream_complete`, `skill_created`,
`skill_refined`, `memory_updated`, `curator_report`.

The bridge subscribes `signal_bus.subscribe("*", self._on_signal)` — so **every** SignalBus
emission is forwarded, including the managed-task `task_completed` / `task_failed` signals that
**coding-mode runs emit** (a coding run is a tasks.db managed task). That is the urgency: full
chat content, tool inputs/outputs, and coding-task lifecycle all stream in clear to any connector.

## Client inventory (this decides the mechanism)

1. **Beacon Desktop (Electron, Node `ws`)** — `src/main/connection/index.ts` connects to `wsUrl`
   (:8010). Comment at index.ts:38 notes the WS is "unauthenticated by the daemon"; index.ts:100
   already has dark plumbing for "a future authenticated :8010". Node `ws` *can* set upgrade
   headers.
2. **Shipped static web UI (BROWSER, native `WebSocket`)** — `src/prometheus/web/static/index.html`
   does `ws = new WebSocket(WS_URL)` ([index.html:512](src/prometheus/web/static/index.html#L512)),
   `WS_URL = ws://<host>:8010`. Served by the daemon's `StaticFiles` mount
   ([server.py:1177](src/prometheus/web/server.py#L1177)). **Browsers cannot set an `Authorization`
   header on the WS upgrade.** (Aside: this UI sends no bearer on its REST calls either, so it is
   already non-functional against the bearer-gated REST — a pre-existing, out-of-scope gap.)
3. **Daemon's own FastAPI** — uses the bridge in-process via `app.state.ws_bridge.dispatch_user_message`
   (`POST /api/chat/send`). This calls Python methods directly; it is **not** a WS client and is
   unaffected by WS auth.

## DECISION: first-frame auth

A real **browser** client exists (#2), and the browser WebSocket API cannot attach an
`Authorization` header to the upgrade. Per the sprint rule, the existence of any browser client
mandates **first-frame auth**, not Bearer-on-upgrade. Chosen mechanism:

- On connect, the server sends **nothing** and waits up to **5s** for the first frame.
- The first frame must be `{"type":"auth","token":"<PROMETHEUS_API_TOKEN>"}`. Token compared with
  `hmac.compare_digest` (constant-time). **No token in query strings, ever** (they leak into logs).
- Success → the normal flow begins (the `connected` welcome is moved to *after* auth, so an
  unauthenticated socket receives **zero** frames).
- Missing / malformed / wrong / late (>5s) → `close(code=4401)`, zero frames delivered.
- **Same token source as REST:** `config.web.api_token` or `PROMETHEUS_API_TOKEN`; **empty token =
  auth disabled** (exact parity with the REST middleware — dev/no-token setups keep working,
  including the tokenless static UI).

Uniform mechanism for all clients (Beacon + browser) — simpler than splitting by transport, and
the browser constraint forces it anyway.

## Clients I must update so I don't break them

- **Static web UI** (this phase): send `{"type":"auth", token}` from a `localStorage` token before
  `subscribe`; show an "unauthorized" WS state on 4401. Token is read client-side, never embedded
  in the served HTML.
- **Beacon WS client** (Phase B): authenticate with the safeStorage token; surface 4401 in the
  unified auth-failure UX.

## Tests (Phase A, `python3 -m pytest`)

Real `websockets` client against a real `WebSocketBridge` (token = a random per-test value, never
the real secret):
1. Unauthenticated connect → closed **4401** and **zero frames** delivered (assert `recv()` raises
   `ConnectionClosed` with no prior message — side-effect, not just "close was called").
2. Wrong-token first frame → closed 4401, zero frames.
3. Malformed first frame (not JSON / not `type:auth`) → closed 4401, zero frames.
4. Late auth (silent >5s) → closed 4401, zero frames.
5. Correct auth → `connected` welcome received, then a broadcast event is delivered normally.
6. Auth disabled (empty token) → legacy behavior preserved (welcome + events, no auth needed).
